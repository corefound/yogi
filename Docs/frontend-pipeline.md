# Frontend Pipeline

The frontend is responsible for reading `.io` source files, resolving project
modules, validating semantics, and writing cache artifacts that the C++ backend
can consume.

## Source Files

Yogi source uses TypeScript-style syntax. The parser produces an AST for each
module. The language keeps a stricter semantic layer on top of that syntax, so
some TypeScript forms are parsed but rejected later.

Example:

```ts
import { addWeight } from "./scoring"

let retries: number = 0
let score: number = addWeight(10, 4)
```

## Module Discovery

The compiler discovers modules starting from the entry file. Imports are
resolved into a dependency graph and then processed in dependency order. The
frontend writes module artifacts into:

```text
packages/.cache/modules/<module-name>/
```

Each module can produce:

- `ast.fb`: FlatBuffer encoded AST.
- `sir.fb`: FlatBuffer encoded Semantic Intermediate Representation.
- `<module>.ll`: LLVM IR generated by the backend.
- `<module>.o`: object file generated from the LLVM module.

The global metadata file is written into:

```text
packages/.cache/meta.fb
```

The C++ compiler driver reads that global metadata file after the frontend
finishes.

## Semantic Analysis

Semantic analysis validates language rules and converts AST nodes into SIR.
Examples of semantic rules:

- Variables must use `let` or `const`.
- Variables must have explicit type annotations.
- Variables must have initializers.
- Assigning `any` into a concrete type requires an explicit cast.
- `const` variables cannot be reassigned.
- Object, array, and tuple assignments are checked against their declared type.
- Imports must reference exported symbols from the imported module.

## FlatBuffer Outputs

Yogi writes both AST and SIR as FlatBuffers. They are intentionally separate:

- AST preserves parser-level structure.
- SIR preserves validated semantic structure used by the backend.

This separation lets the compiler keep source syntax, semantic meaning, and
native lowering as different phases.
